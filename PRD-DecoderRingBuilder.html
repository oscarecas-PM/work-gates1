1. Purpose & Background

EXPLICIT The Gate Decoder Ring Builder is a companion tool to the Gate Tracker application. It provides a visual, drag-and-drop interface for assigning manufacturing operations to numbered gates, then exporting the result as a Gate Decoder Ring CSV file.

EXPLICIT The Decoder Ring is currently maintained by hand—a tedious, error-prone process. This tool replaces manual CSV editing with a visual workflow: the user uploads an MES operation listing for a single part number, drags operations into gate groupings, assigns gate numbers from a dropdown, and exports a properly formatted CSV that can be pasted into the master Decoder Ring file.

Cross-reference: The output CSV format is defined in PRD-GateTracker.html, Section 3.1 “Gate Decoder Ring (CSV)”. The Gate Tracker application consumes this file as one of its primary data inputs.

1.1 Intended Users

EXPLICIT Primary: Lab leadership and operations planners who define which operations belong to which gates for a given part number.

1.2 Relationship to Gate Tracker

EXPLICIT This tool produces the Decoder Ring CSV; the Gate Tracker consumes it. The two tools share:

  • The Decoder Ring CSV schema (Part No., Part Name, Gate Number, Gate Name, Oper No., Work Dept, Modifier)
  • The Gate Technician CSV format (Gate Number, Gate Name, Tech Name, Priority)
  • Shared utility functions (CSV parser, XSS escaping, column lookup)
  • Visual design conventions (color palette, header styles, card styling)

2. Scope & Constraints

IDSourceRequirement
C-01 EXPLICIT Single self-contained HTML file. No external libraries, frameworks, CDN links, or build steps.
C-02 EXPLICIT All processing is client-side. No data leaves the browser. No server, no network calls.
C-03 EXPLICIT No persistent browser storage (no cookies, localStorage, IndexedDB). State is preserved only via explicit snapshot export.
C-04 ASSUMED Target browsers: Google Chrome and Microsoft Edge (latest versions).
C-05 EXPLICIT Operates on one part number at a time. The user builds a gate mapping for a single part, exports it, then starts fresh for the next part.
C-06 EXPLICIT The exported CSV is designed to be appended to a larger master Decoder Ring file that contains mappings for multiple parts.

3. Data Inputs

3.1 Gate Technician CSV (Optional)

EXPLICIT The same Gate Technician file used by the Gate Tracker. Provides the list of available gates for the dropdown.

ColumnRequiredDescription
Gate NumberYesNumeric gate identifier. Gates ≥1 have fixed names; gate 0 is a catch-all for work outside the lab.
Gate NameYesDescriptive gate name. The gate number–to–name pairing is extracted from this file and used to populate dropdowns.
Tech NameYesTechnician name (used by Gate Tracker but not by this tool directly).
PriorityYesTechnician priority within the gate (used by Gate Tracker but not by this tool directly).

EXPLICIT The gate number–to–name mapping is repeated across multiple rows (one per technician). This tool deduplicates to build a unique mapping. The user expressed a desire to define this pairing only once; however, adding a separate file was rejected to minimize the number of input files.

ASSUMED If no Gate Technician CSV is uploaded, the user can still create gates on the fly via the “New Gate” option in the dropdown.

3.2 MES Operations CSV

EXPLICIT A downloaded listing of all operations for a single work order from the Manufacturing Execution System.

Oper TitleYesYesDescriptive operation name. Displayed on the card face.
Work DeptYesYesDepartment code. Drives card color and is carried through to the exported CSV.
Oper TypeNoTooltipOperation type (e.g., LABOR, INSPECT). Shown in tooltip.
Labor Hours Per UnitNoTooltipEstimated labor hours. Shown in tooltip.
Earnable Hours FlagNoTooltipWhether hours are earnable. Shown in tooltip.
Additional columns (Oper Status, Optional Operation, Oper Hold Status, Actual Start/End Date, Updt Userid, Work Center, Sched Start/End Date, Work Loc, Process type, Skill Category, Crew Qty, Building/Room, LAC, Oper Network, Labor Hours Setup, Process Hours, Labor Hours Inspect, Total Time) are present in the file but ignored by this tool.

3.3 Part Information (Manual Entry)

EXPLICIT The user manually enters the Part Number and Part Name. Part Number is required; Part Name is optional but recommended.

4. Functional Requirements

4.1 File Loading

IDSourceRequirement
F-01 EXPLICIT The Gate Technician CSV is loaded first (optional). The application extracts unique gate number–to–name pairs and populates the gate assignment dropdown.
F-02 EXPLICIT The user enters Part Number (required) and Part Name (optional) in text fields.
F-03 EXPLICIT The MES Operations CSV is loaded and parsed. Required columns: Oper No, Oper Title, Work Dept. Optional columns: Oper Type, Labor Hours Per Unit, Earnable Hours Flag.
F-04 ASSUMED The “Load Operations” button is disabled until Part Number is entered and MES Operations CSV is loaded. Clicking it parses the CSV and renders the workspace.
F-05 ASSUMED Operations with Oper No of “0” or blank are skipped during parsing.

4.2 Workspace & Card Display

IDSourceRequirement
F-10 EXPLICIT After loading, each operation is displayed as a card within its own gate outline (a dashed-border container). Gate outlines are arranged horizontally, left-to-right, in ascending Oper No order.
F-11 EXPLICIT Each card displays: the Oper No (prominent) and Oper Title (smaller, below).
F-12 EXPLICIT Each card has a tooltip showing: Oper No, Oper Title, Work Dept, Labor Hours Per Unit, Earnable Hours Flag, and Oper Type (when available).
F-13 EXPLICIT Card color is determined by Work Dept:
  • Green (#16a34a) — Work Dept equals “SMART”
  • Blue (#2563eb) — Work Dept ends with “_QA”
  • Pink (#db2777) — All other Work Dept values
F-14 ASSUMED Within any gate outline, operations are always sorted in ascending Oper No order (numerically). This sort is enforced after every drag-and-drop action.

4.3 Gate Assignment Dropdown

IDSourceRequirement
F-20 EXPLICIT Each gate outline has a dropdown in its header to assign a gate number. The dropdown is populated with:
  1. “(select gate)” — default, unassigned
  2. All unique gate entries from the Gate Technician CSV, formatted as “Gate N: Name”
  3. “+ New Gate…” — for creating a gate not in the CSV
F-21 EXPLICIT Selecting “+ New Gate…” prompts the user for a new gate number and gate name. The new gate is added to the dropdown for future use within the session.
F-22 ASSUMED When a gate outline is assigned a gate number, its border changes from dashed to solid, and a label (“Gate N: Name”) appears above the dropdown.
F-23 ASSUMED Multiple gate outlines may be assigned the same gate number without merging. This supports gate re-entry patterns (e.g., gate 3 → 5 → 6 → 3 again). Only dragging operations into another outline combines them; dropdown assignment never merges outlines.

4.4 Drag and Drop

IDSourceRequirement
F-30 EXPLICIT Operation cards are draggable. Dragging a card into another gate outline moves the operation to that gate. The gate re-sorts by Oper No after the drop.
F-31 EXPLICIT If a drag operation leaves a gate outline empty, that gate outline is removed from the workspace.
F-32 EXPLICIT Dragging a card into the space between two gate outlines creates a new, unassigned gate outline at that position containing the dropped operation(s).
F-33 EXPLICIT Multi-select: Clicking a card selects it (highlighted border). Ctrl+click toggles additional cards into the selection. Dragging any selected card moves all selected cards together.
F-34 ASSUMED When dragging multiple cards, a ghost label shows the count (e.g., “3 operations”).
F-35 ASSUMED Visual feedback during drag: gate outlines highlight with a blue border when a card is dragged over them; drop zones between gates show a blue bar.
F-36 ASSUMED Selection is cleared after a successful drop.
F-37 ASSUMED Clicking a selected card (without Ctrl) deselects all. Clicking an unselected card (without Ctrl) selects only that card.

4.5 Modifiers

IDSourceRequirement
F-40 EXPLICIT Each operation card has a small modifier dropdown with options: blank (no modifier) and “ignore”.
F-41 ASSUMED When modifier is set to “ignore”, the card is visually dimmed (50% opacity) and the Oper Title is struck through.
F-42 ASSUMED The modifier value is included in the exported Decoder Ring CSV Modifier column.
Cross-reference: The “ignore” modifier is defined in PRD-GateTracker.html, Section 3.1. When the Gate Tracker processes a Decoder Ring with this modifier, it skips the operation when determining which gate a work order belongs on.

4.5.1 Omit from Export

IDSourceRequirement
F-43 EXPLICIT Each operation card has an omit button (×) that appears on hover in the top-right corner. Clicking it toggles the operation’s omitted state.
F-44 EXPLICIT Omitted operations remain visible on screen (not removed) so the user can review and undo the action. They are visually distinguished with reduced opacity (0.35), strikethrough on the title, and a diagonal-stripe overlay.
F-45 EXPLICIT Clicking the omit button on an already-omitted operation un-omits it, restoring it to normal appearance and re-including it in the CSV export.
F-46 EXPLICIT Omitted operations are excluded from the Decoder Ring CSV export. They are still included in the saved HTML snapshot state so they can be un-omitted later.
F-47 ASSUMED Omitted operations remain draggable, selectable, and editable (modifier dropdown still functional). Omit is independent of the modifier field.

4.6 Export

IDSourceRequirement
F-50 EXPLICIT Export Decoder Ring CSV: Generates a CSV file with columns: Part No., Part Name, Gate Number, Gate Name, Oper No., Work Dept, Modifier. Rows follow the workspace left-to-right order (not sorted by gate number), preserving gate re-entry sequences. Within each gate outline, operations are sorted by Oper No ascending. The same gate number may appear multiple times in the output.
F-51 ASSUMED If any gate outlines have no gate number assigned, the user is warned. Operations in unassigned gates are skipped in the export.
F-52 ASSUMED Filename format: decoder-ring-{partNo}-{YYYY-MM-DD}.csv. Special characters in the part number are replaced with underscores.
F-53 EXPLICIT The exported CSV rows are designed to be copied into a larger master Decoder Ring file that contains mappings for all part numbers.
F-60 EXPLICIT Save for Editing: Exports a self-contained HTML snapshot file that preserves all current state (part info, gate map, operations, gate assignments, modifiers). Opening the snapshot restores the full editing workspace.
F-61 ASSUMED The snapshot includes: all CSS, all JavaScript functions (serialized via Function.toString()), state as sanitized JSON, and the proprietary banner.
F-62 ASSUMED When opened, the snapshot hides file input fields (data is already embedded) but provides a Gate Technician CSV upload option (collapsed by default) so the user can add new gate definitions.
F-63 ASSUMED The user can export CSV and save further snapshots from within a loaded snapshot, enabling iterative refinement without starting from scratch.

6. Data & Security

IDSourceRequirement
S-01 ASSUMED XSS prevention: all user-entered and CSV-sourced text is escaped via the esc() function (5-character escaping: &, <, >, ", ') before insertion into HTML.
S-02 ASSUMED CSP meta tag: default-src 'self' 'unsafe-inline' included in both the tool and snapshot files.
S-03 ASSUMED CSV field length limit: 500 characters maximum per field, enforced via csvField().
S-04 ASSUMED JSON sanitization in snapshots: all < characters in JSON strings are replaced with \u003c to prevent script tag breakout.
S-05 ASSUMED CSS.escape() used for querySelector calls with dynamic operation numbers.

7. Open Questions

#StatusQuestion
1 OPEN Multi-part sessions: Should the tool support building gate mappings for multiple part numbers in a single session (e.g., load part A, export, load part B, export), or is the current one-at-a-time-then-reload approach sufficient?
2 OPEN Undo/Redo: Should the tool support undoing drag-and-drop actions? Currently, the only way to undo is to manually drag cards back.
3 RESOLVED Gate ordering in export: The export preserves left-to-right workspace order, not gate-number sort order. This is required to support gate re-entry patterns (e.g., gate 3 → 5 → 6 → 3). The Gate Tracker PRD confirms “row order dictates left-to-right display order of gates.”
4 OPEN Additional modifiers: Are there modifier values beyond “ignore” that should be supported? The Gate Tracker PRD notes “other values are reserved for future use.”
5 FUTURE Import existing Decoder Ring: Could the tool import an existing Decoder Ring CSV to edit previously defined gate mappings, rather than only creating new ones?
6 FUTURE Gate number–to–name single source of truth: The user noted frustration with repeating gate number/name pairings in the Tech CSV. A future enhancement could allow a dedicated gate definition file or a header section in the Tech CSV for unique gate definitions.

8. Architecture Notes

8.1 Shared Utilities

The following functions are copied from gate-tracker-18.html to maintain consistency and enable future extraction into a shared utility module:

  • parseCSV(text) — RFC 4180 compliant CSV parser
  • csvField(s) — Field length limiter (500 chars)
  • colIndex(headers, name) — Case-insensitive column index lookup
  • loadCSV(file, callback, statusId) — FileReader wrapper with error handling
  • esc(s) — XSS-safe HTML escaping
  • normOper(s) — Leading-zero stripping for operation numbers

8.2 State Model

Application state consists of five globals:

  • partInfo — { partNo, partName }
  • gateMap — { gateNumber: gateName } from the Tech CSV + user-created gates
  • operations — Array of operation objects, each with a gateId linking it to a gate outline
  • gates — Ordered array of gate outline objects { id, gateNumber, gateName }
  • selectedOps — Set of selected operation numbers for multi-select

8.3 Drag-and-Drop Implementation

Uses the HTML5 native Drag and Drop API (dragstart, dragover, dragleave, drop events). No external library required. Drop targets include gate outlines (for “move into”) and thin drop zones between outlines (for “create new gate”).

8.4 Snapshot Serialization

The “Save for Editing” feature creates a self-contained HTML file following the same pattern as Gate Tracker snapshots: CSS inlined in a <style> tag, state embedded as sanitized JSON in a <script> tag, and all JavaScript functions serialized via Function.toString(). On load, the snapshot reads the embedded state and calls render() to rebuild the interactive workspace.
