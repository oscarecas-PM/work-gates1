Product Requirements Document (PRD)

Product Name: Employee Score Card

File: employee-score-card-1.html

Version 1.7 | May 1, 2026 | Author: Oscar Castillo

1. Overview

A client-side web application that joins per-employee labor charges (xlsx exports from the labor-charge system) with the Hours Earned CSV (already produced for Earned Hour Analysis) to compute per-employee, per-month earned hours, actual hours, and CPI. When multiple employees charge the same operation, earned credit is split pro-rata by each employee's share of total charged hours on that operation. The tool surfaces both raw productivity (earned hours per technician) and efficiency (CPI), supports multiple distribution views to characterize disparity, and produces a ranked list of part numbers whose actual charges are not yet matched to earned hours so the user can prioritize follow-up exports.

2. Objectives

  • Ingest one or many .xlsx employee charge files (multi-file because the source system limits exports by part-number wildcard) and dedup overlapping rows
  • Convert cumulative Total Hours values to incremental per-week charges per (Employee, Order No, Operation)
  • Join charges to the Hours Earned CSV on (Order No, Oper No) and split each closed operation's earned hours pro-rata across the employees who charged on it
  • Aggregate per-employee per-month: Total Earned Hours, Total Actual Hours, CPI
  • Visualize the distribution of productivity across employees with multiple chart perspectives (ranked bar, histogram, monthly box plot, actual-vs-earned scatter)
  • Produce a per-month grid (employee × month) with drill-down to contributing operations, including a "split" flag for operations shared across techs
  • Surface unmatched charges (charge rows whose operation has no Hours Earned row) ranked by total actual hours so the user can prioritize follow-up data pulls
  • Provide an anonymize toggle that replaces names/badges with stable Tech 1, Tech 2, … labels throughout the UI, exports, and snapshots
  • Provide self-contained HTML snapshot, CSV exports, and Print/PDF output

3. Architecture Constraints

  • Entirely client-side; all parsing, joining, splitting, and chart rendering happens in the browser
  • Single self-contained HTML file with embedded libraries (no CDN, no network calls)
  • No persistent storage (cookies, localStorage, etc.). All data resides in memory for the session
  • Must work in modern browsers (Chrome, Edge)
  • Handle multi-file xlsx ingest of tens of thousands of rows without freezing
  • Clean, professional UI styling matching the existing tool family (navy/gray palette, right-aligned numerics)
  • Personally identifiable information (employee name, badge) never leaves the browser

5. Data Inputs

5.1 Employee Charge Files (.xlsx, required, multiple)

The user uploads one or more .xlsx files. Each file's data table starts on row 4 (rows 1–3 are export metadata and are skipped). The header row is auto-detected by scanning the first 10 rows for a cell whose value matches Order No, mirroring the auto-detect approach in gate-tracker-30.html. Header matching is normalized via normHeader() — case-insensitive and tolerant of routine punctuation/whitespace variations (e.g. Order No., order_no, Order No all match). The same normalization is applied to required-column lookups via colIndex(), eliminating the trailing-period ambiguity in headers like Part No. vs Part No.

ColumnTypeNotes

Week EndingDate/TimeFormat MM/DD/YYYY HH:MM:SS AM/PM. Used to attribute Actual Hours to a calendar month and as part of the dedup key.
Employee NameStringDisplay label only. Names may repeat or change — not used as identity.

CCCString (optional, v1.7)Cost-center / category code used to bulk-exclude employee groups (e.g., quality inspectors). Surfaces as the “Exclude CCC” chip strip when present in any uploaded file.

Multi-file ingest: rows are concatenated across files. Duplicates by composite key (Badge, Order No, Oper No, Week Ending) are deduplicated, keeping the row with the highest Total Hours (most recent snapshot wins).

Optional columns: CCC and Work Center are looked up via colIndex() and captured per row when present; files without these columns ingest exactly as before, and the corresponding filter chip strips are simply hidden.

5.2 Hours Earned CSV (required)

Identical format to PRD-EarnedHourAnalysis.html §5.1. Required columns: Order No, Year of Actual End Date, Month of Actual End Date, Oper No, Hours Earned. The optional Actual End Date column is preserved but not required.

Each row represents one closed operation. The Earned Hour Analysis tool aggregates these rows to (Order No, Month) for its bundle view; this tool keeps them at (Order No, Oper No, Month) grain so per-operation pro-rata splits are possible.

5.3 EV Details CSV (optional, multiple files)

Identical format to PRD-EarnedHourAnalysis.html §5.3. Used solely for Program enrichment of the Program filter chips. Columns: Network, Order No, Part No, Plan Title, Program. Duplicate Order Nos deduplicated; first occurrence wins. If not uploaded, the Program filter chips are hidden.

6. Functional Requirements

6.1 Data Processing

F-01 Auto-detect the header row in each xlsx by scanning the first 10 rows for a cell whose normalized value matches Order No (case-insensitive, punctuation- and whitespace-tolerant via normHeader()). Data starts at the row after the detected header. Files where no header is found produce a per-file failure recorded in the validation panel; remaining files in the batch continue to parse independently (see F-02a).

F-01a Required-column lookups via colIndex() use the same normHeader() normalization, so headers like Part No vs Part No., or Total Hours vs Total Hours, all resolve to the same column.

F-02 After concatenating rows from all uploaded xlsx files, deduplicate by (Badge, Order No, Oper No, Week Ending). When duplicates exist, retain the row with the maximum Total Hours. Surface dedup count in the validation panel.

F-02a Per-file resilience. The charge-file ingest loop processes each .xlsx independently. A failure in one file (missing header, missing required columns, read error) does not abort the batch — remaining files continue to parse. The status line shows N of M file(s) parsed, and failed files are listed by name in both the status line and the validation panel (with the underlying error message). Analyze remains enabled as long as at least one charge file parsed successfully and the Hours Earned CSV is loaded.

F-03 Convert cumulative Total Hours to incremental per-week charges. For each (Badge, Order No, Oper No) series, sort by Week Ending ascending and compute incremental[i] = Total Hours[i] - Total Hours[i-1] (incremental[0] = Total Hours[0]). Negative diffs (corrections / re-categorizations upstream) are clamped to 0 and counted in the validation panel.

F-04 Parse the Hours Earned CSV with the same column-tolerance and parseMonth() logic used in earned-hour-analysis-1.html. Non-numeric Hours Earned values treated as 0 with a warning.

F-05 Pro-rata split. For each row in the Hours Earned CSV (one closed operation = one earned-hours value): total_charged = Σ incremental_charges over all badges on (Order No, Oper No)
earned_share[badge] = Hours Earned × (charge[badge] / total_charged)

  • If total_charged === 0: credit nothing; add the operation to the "earned but unmatched to charges" warning list.
  • If more than one badge contributed charges: flag the operation as split for the per-month drill-down view.
  • The earned_share is credited to the calendar month identified by (Year of Actual End Date, Month of Actual End Date) for that Hours Earned row.

F-06 Per-employee per-month aggregation.

  • Total Earned Hours = Σ earned_share across all operations that closed in (Badge, YearMonth).
  • Total Actual Hours = Σ incremental charges where Week Ending falls in YearMonth, regardless of operation status. Both CLOSE and ACTIVE charge rows count.
  • CPI = Earned / Actual. Display as em-dash (—) when Actual = 0.

F-07 Active operations contribute zero earned hours regardless of how many charges sit against them. They contribute fully to Actual Hours. Operations transition to earned-hours contributors only when they appear in the Hours Earned CSV.

6.2 Filters

F-10 "Data starts from" filter: Month/Year dropdowns. Default: January 2026. Months before the selected month are excluded from all summary tables, charts, and the per-month grid.

F-11 "Hide current month" toggle: Checkbox, checked by default. Removes the real-world current calendar month (today's month) from all views, on the assumption that the live month is always partial / in progress. Note: this is the calendar current month, not necessarily the latest month present in the data — for a snapshot of past data those can differ. (Reworded in v1.7 to match the implementation.)

F-12 Program filter chips: Visible only when EV Details CSV is uploaded. Clickable chips for each unique Program. Blue = included, grey strikethrough = excluded. Deselecting a program removes its WOs everywhere (summary, charts, grid, exports).

F-13 Employee multi-select (renamed from "Badges" in v1.7): Full-width chip strip listing every employee discovered in the charge files, sorted alphabetically by employee name (badge fallback when name is missing or when Anonymize is ON). Default = all included. Clicking a chip flips its effective state and records a per-badge override that wins over CCC / Work Center auto-exclusion. All / None buttons set or clear overrides for everyone.

F-13a CCC exclude chips (v1.7). Visible only when at least one uploaded charge file carries the optional CCC column. Multi-select; clicking a chip toggles membership in State.filters.excludedCCCs. Any employee with a matching CCC is auto-excluded; their Employee chip flips to the OFF style so the user can visually verify the bulk action.

F-13b Work Center exclude chips (v1.7). Same model as F-13a, gated on the presence of the Work Center column. Composes with the CCC filter (logical OR — an employee is auto-excluded if they match either filter on any of their charge rows).

F-13c Auto vs. manual exclusion (v1.7). An Employee chip's displayed state reflects effectiveOn(badge), computed as: manual override (if set) wins; otherwise follow the auto rule (excluded if any CCC/WC match). Clicking a chip writes the opposite of its current effective state into State.filters.manualOverride; if the new state happens to match the auto state, the override is cleared so the chip resumes following auto.

F-14 "Anonymize" toggle: Lives in the top scalar-controls row alongside Start month, Hide current month, and Hypothetical Close (v1.7 layout). When ON, all visible badges and names are replaced with Tech 1, Tech 2, … labels. Mapping is deterministic for a given session: badges are sorted alphabetically and assigned indices in that order so the same badge always becomes the same Tech N within a session.

F-15 Filters apply immediately on toggle/click (no separate Apply button), with a debounce on numeric range changes.

F-15a "Hypothetical close" toggle (v1.7). Checkbox in the top scalar-controls row, default OFF. When ON, every charge against an operation that has no matching Hours Earned row is treated as if the operation closed at CPI 1.0: its incremental hours are added to both Earned and Actual for the corresponding (badge, month) cell. The Unmatched Charges section is unaffected (it remains the data-quality view). A persistent amber banner above the results announces the assumption while the toggle is on. Toggle OFF restores real CPI.

6.3 Summary Section

F-20 Summary section at top, titled "Per-Employee Totals (Filtered Range)". Rendered as a collapsible <details> element, collapsed by default in both the live tool and the saved snapshot. Header shows a rotating arrow indicator (▶ closed, ▼ open). Print mode forces it open via @media print CSS so PDF exports always include the table.

F-21 One row per badge with columns: Tech (badge or anonymized label), Earned Hours, Actual Hours, CPI, # Split Operations participated in, # WOs touched. Footer row shows column totals (CPI footer = sum(Earned) / sum(Actual)).

F-22 Sortable on every numeric column (click header to toggle ascending/descending). Default sort: Earned Hours descending.

F-23 Tech labels in the first column hyperlink to the badge's row in the per-month grid.

F-24 Color-coded CPI cells: green for ≥ 1.0, amber for 0.7–1.0, red for < 0.7. Em-dash cells (Actual = 0) are uncolored.

6.4 Distribution Charts

F-30 Distribution panel with a tab strip exposing four sub-views. Only one tab is visible at a time on screen; all four are exported in the snapshot and printed.

F-31 Tab A — Ranked Bar: Two bars per badge (paired side-by-side), sorted by Total Earned Hours descending. Orange bar = avg earned hours per active month (left Y-axis, primary), where "active month" means a month in which the badge had charged hours > 0. Blue bar = total earned in the filtered range (right Y-axis, secondary). Two horizontal dashed reference lines drawn against the orange axis: "Group avg" (workforce-wide average of per-month rates) and "Middle tech" (median per-month rate — half of techs earn more, half earn less). Audience-friendly labels intentionally avoid the bare "mean"/"median" stats jargon. Bar tooltip footer always defines both reference lines with their numeric values for hover discoverability. Subtitle line under the title makes the dual-axis layout explicit. Legend at top; axis titles and tick labels color-coded to each series. The orange (per-month rate) series is the input for the future ECD-from-distribution work in Gate Tracker (per §10). v1.7 restyle: both reference lines and their labels render in black (was green / amber, now reserved for accept-vs-reject signals); each label sits over a rounded semi-transparent white plate; labels are right-justified and staggered — whichever line is higher up on the chart has its label shifted further to the left, the lower one's label sits closer to the right edge, so they never overlap each other or the bars beneath them.

F-32 Tab B — Histogram: Bins the per-tech "average earned hours per active month" metric (same calculation as F-31's orange series). Bin width via Freedman–Diaconis rule (2 × IQR × n^(-1/3)) with a minimum of 1 hour per bin (averages are typically much smaller than total earned over a multi-month range). Bar height = count of badges in bin. Hovering a bar shows the badge labels in that bin (limited to 10 with "+N more" overflow). X-axis: avg earned hrs/active month bins. Y-axis: # employees. Subtitle line under the title makes the metric explicit.

F-33 Tab C — Box Plot per Month: One vertical box per month in the filtered range, computed from the per-(badge, month) Earned Hours values. Box = Q1–Q3 (middle 50% of techs); horizontal line in box = median; whiskers = min/max within 1.5×IQR; red dots = outliers; n=N label centered below each box's lower whisker = number of techs that month. Drawn on a Chart.js canvas via custom plugin (no extra dependency). The X-axis uses a category scale so month labels render correctly. Above the chart: a purpose-explainer paragraph (connecting distribution width to production-rate uncertainty) and a legend strip with colored swatches explaining the box, median line, whiskers, outliers, and n. Hovering anywhere within a box's column shows a tooltip with all stats: n techs / median / middle 50% (Q1–Q3, IQR) / whiskers / outlier count. Outlier dots retain their per-point tooltip.

F-33a "Ignore outliers" toggle (v1.7). Checkbox above the legend strip. When ON, the red outlier dots are hidden and the Y axis is recomputed without the outlier values, so the boxes & whiskers expand to fill the available vertical space. Box statistics (Q1, median, Q3, whiskers, IQR) are unaffected — the toggle is purely visual, addressing the case where one or two extreme outliers compress all the boxes against the X axis.

F-34 Tab D — Scatter (Actual vs. Earned): One dot per badge. X = Total Actual Hours, Y = Total Earned Hours, both over the filtered range. Dashed diagonal at y = x represents CPI = 1.0. The "CPI = 1.0" label sits below-left of the diagonal endpoint, in bold, offset further to the left so it doesn't overlap the line itself (v1.7). Hovering a dot shows the badge label, Earned, Actual, and CPI.

F-35 Each chart respects the Anonymize toggle: when ON, axis labels and tooltips display Tech N rather than badge / name.

F-36 Each chart respects the Badge filter: deselected badges are hidden from the rendered chart.

6.5 Per-Month Grid

F-40 Grid with badges as rows (sorted by total Earned Hours descending) and months as columns (oldest to newest in the filtered range). Rendered as a collapsible <details> element, collapsed by default in both the live tool and the saved snapshot (matches F-20). The Tech column is a stacked cell — technician name on top (bold), badge below in small dim text — reducing column width vs. the prior single-line Name (Badge) format. When Anonymize is ON, only the Tech N label shows (no badge sub-line). The Tech column is sticky-left and the header row is sticky-top, both stay visible while the grid scrolls horizontally and the page scrolls vertically; the corner cell sits above both. v1.7 scroll-control update: the ◄ / ► scroll buttons (one click ≈ one month-column width) now sit side-by-side in a thin pill at the top-right of the grid section, pinned via position: sticky; top: 8px; so both arrows remain reachable as the user scrolls vertically through a tall section. Buttons are hidden when the grid does not overflow horizontally and disable at scroll extremes; visibility is re-evaluated on the section's toggle event so the controls appear correctly on first open.

F-41 Each cell shows three stacked values, each with a small uppercase label suffix for legibility: 9.1 EARNED (top, bold), 22.7 ACTUALS (middle, smaller dim font), 0.40 CPI (bottom, color-coded by F-24 thresholds). The label suffix is a small caps slate-400 span (.lbl) that follows each number. The rightmost Total column uses the same labeled three-value format. v1.7: the per-cell “N split” pill is removed — split participation is auxiliary information for this view; users still see the SPLIT badge in the drill-down modal (F-42) when they need it.

F-42 Cells are clickable. Clicking opens a drill-down modal that opens with a CPI summary block at the top (Earned, Actuals, CPI as large stat tiles plus the explicit calculation {earned} ÷ {actuals} = {cpi}), followed by a table listing every operation that contributed to that (badge, month) cell. Operation table columns: Order No, Oper No, Part No., Description, Earned (op), Total Charged (op), My Charged, My Share, Flag. A second table below lists all the badge's incremental charges in the month (matched and unmatched).

F-43 The drill-down modal includes a "Copy as CSV" button.

F-44 Empty cells (no activity) render as a faint dash, not zero.

F-52 "Export Unmatched CSV" button next to the section header. CSV includes all visible columns plus one extra section listing the underlying (Order No, Oper No) tuples and their actual hours.

F-53 Filename: employee-score-card-unmatched-YYYY-MM-DD.csv.

F-54 The total at the bottom shows the sum of unmatched actual hours and the percentage of all charged hours represented by the unmatched bucket. This communicates the total earned-hours blind spot.

  5. ECD-from-distribution (deferred to Gate Tracker, agreed 2026-04-30): The Box Plot per Month was built to support estimated-completion-date forecasting from per-tech production-rate distributions. That feature will live in Gate Tracker (where techs and managers watch weekly), not in the Score Card. Methodology agreed: backward = solve "equivalent heads" from a bundle's earned hours and the median per-tech rate (translates "5 names touched the WO" into "but only ~2.3 equivalent full-time heads worth of work happened"); forward = apply assumed headcount × per-tech rate (low/median/high from quartile presets, default low=p25 / median=p50 / high=p75 with override) to remaining bundle hours to produce ECD(early)/ECD(median)/ECD(late). The Score Card may need a CSV export of the per-tech-per-month rate distribution as a Gate Tracker input when that feature is built. Open: file format and join key.

11.2 Historical Employee Charge data sourcing

The tool's primary input is Employee Charge .xlsx exports (per-employee, per-operation, per-week, cumulative Total Hours). This schema is materially different from any input we ingest elsewhere:

  • P Files: WO-level cumulative actuals at a snapshot date. No employee breakdown. Cannot substitute.
  • VAC Tracker Log: WO-level cumulative VAC, Actual Hours, EV, and CPI per snapshot date. No employee breakdown. Cannot substitute.
  • Hours Earned CSV (existing): per-WO-per-operation closure events. Provides the numerator (earned hours per closed operation) but no charge data. The tool already requires this and uses it correctly.

Two assumptions tracked for production-readiness:

  1. Current data availability — RESOLVED (2026-04-30, reaffirmed in v1.7). The labor-charge system produces the .xlsx schema described in §5.1: v1.2 ingested 11 real exports successfully, and v1.7 confirms the additional optional CCC and Work Center columns are also present in real exports. The input contract is no longer hypothetical.
  2. Historical data availability — OPEN. Whether the labor system retains exportable per-employee history beyond a rolling snapshot window is still unverified. Operational depth (how many months of trended CPI we can show) depends on this answer.

If retention turns out to be too short, the path forward is one of:

  • Begin archiving weekly .xlsx exports going forward to build up depth over time.
  • Timecard / payroll system direct export (different report, different schema — would require adapting §5.1).
  • Per-operation labor allocation derived from MES audit history + technician assignments (much weaker signal; may not preserve charge-hour fidelity).
  • Limit the tool to current-only views and live with the lack of historical depth.

The tool is no longer gated on this question for current-month / current-quarter use. Historical depth simply scales with how far back exports are available.
