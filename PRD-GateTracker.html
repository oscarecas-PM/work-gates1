1. Purpose & Background

EXPLICIT The Gate Tracker is a client-side, single-file HTML application that visualizes where active work orders sit within a manufacturing gate sequence. It ingests two CSV data sources—a Gate Decoder Ring (the master gate/operation mapping) and a Daily Upload (a daily snapshot of operational status)—and renders a dashboard showing order-level progress through gates.

EXPLICIT The application must run entirely offline in a standard web browser with zero external dependencies. The user cannot share work data with external services, so all processing is done client-side. The HTML file is transferred to a work computer via download.

RESOLVED The lab is converting to a visual factory model: physical racks represent predefined gates, and hardware (when ready to work) is placed on the appropriate rack based on which operations are in queue or active. Executing this in the real world is challenging, so this tool provides a quick digital reference showing what operations are in queue/active for each work order and which rack (gate) the hardware belongs on. Visibility into work order or operation holds is critical—held hardware should be moved from ready-to-work gates to a dedicated hold rack.

1.1 Intended Users

RESOLVED Primary: Lab leadership who need to quickly ascertain where hardware should be physically placed. Secondary: Project managers who want to understand the week-to-week progress of their hardware through the gate sequence.

1.2 Business Context

RESOLVED The Decoder Ring is currently maintained manually. A more efficient approach would be to assign gate information for every operation directly within the Manufacturing Execution System (MES), making the manual file a temporary measure. The Daily Upload is generated from the MES but is manually downloaded—also a temporary approach.

This application is a prototype. With funding, it would be rebuilt within the organization's network and connected directly to the MES via API instead of relying on manually downloaded files. The product owner is open to structuring the current code in a way that would make the swap to an API configuration easier for a human programmer in the future.

OPEN To support the eventual API migration, consider: (1) isolating all CSV-parsing logic behind a data-adapter interface so a future developer can swap in an API fetch without touching rendering code, and (2) documenting the expected data schema independently of the CSV format.

4.7 Gate View

EXPLICIT The Gate View is the primary view of the dashboard. It answers the question: "Which physical rack (gate) should each order's hardware be placed on right now?" The Gate View is displayed at the top of the page, above the Detail View.

IDSourceRequirement
F-60 EXPLICIT Single-gate placement (legacy algorithm): Superseded by F-95 when an Audit CSV is loaded. Each order is assigned to exactly one gate. The algorithm walks the decoder ring for that order's Part No. in CSV order and selects the first gate that contains a non-ignored Active or In Queue operation. "First" means the earliest gate in the decoder ring's row order, not the lowest gate number.
F-61 EXPLICIT Modifier "ignore": If the Modifier column in the decoder ring contains "ignore" (case-insensitive) for an operation, that operation is skipped during gate placement. The order will not be placed on a gate solely because of an ignored operation.
F-62 EXPLICIT Hold Rack: An order is placed on the Hold Rack (displayed first, dark red header) if any of the following are true: (1) Order Hold Status is "ORDER STOP" or "ORDER_HOLD" (case-insensitive), (2) the order has no Active or In Queue operations, (3) no qualifying (non-ignored) gate is found. When an Audit CSV is loaded, orders with gate value "HOLD" (case-insensitive) are placed on the Hold Rack. The display label is “Hold Rack” everywhere: Gate View header, Detail View audit location, multi-gate warnings, and mobile audit UI.
F-63 EXPLICIT "Oper stop" handling: If Order Hold Status is "OPER STOP" or "OPER_HOLD" (or any value other than "ORDER STOP"/"ORDER_HOLD"), the order stays on its qualifying gate (not moved to Hold). The hold status is displayed on the card.
F-64 EXPLICIT FIFO ordering: Within each gate, orders are sorted by max Total Days at Operation descending (highest days on the left). This gives a FIFO approximation—the order that has been waiting the longest is positioned first.
F-65 EXPLICIT Gate ordering: Gates are displayed in this order: Hold Rack first, then numbered gates in ascending gate number order, then the “Not found on a rack” section last (see F-96).
F-66 EXPLICIT Program-colored cards (deterministic): Each gate view card is colored by program using a deterministic mapping based on the first letter of the program name: "P" → light blue, "N" → pink. All other first letters fall back to an ordered palette (amber, green, indigo, yellow). This ensures colors are consistent across runs regardless of sort order.
F-67 EXPLICIT Card content: Each card shows Order No., Part No. (with Part Name in parentheses), S/N, hold status (if present, truncated to 12 chars), and Total Days (max across active ops, rounded to 1 decimal).
F-68 EXPLICIT Card tooltip: Hovering shows: Program, Hold status (if any), Total Days with highest-days non-ignored oper number, and a list of all Active/In Queue operations with oper number, title, status, work dept, days at operation, and [ignore] label for ignored operations.
F-69 EXPLICIT Card click → Detail View: Clicking a gate view card selects it (dark border highlight), opens the Detail View if hidden, resets all filters to show all, and scrolls to the corresponding order card in the Detail View. Multiple cards can be selected simultaneously (toggle on/off).
F-85 EXPLICIT Ignore-aware avg days: The avg days value used for FIFO sorting is sourced from the Avg. DaysCurrentOper column. The max value excludes operations marked with the “ignore” modifier. Only non-ignored Active/In Queue operations contribute to the max calculation. Note: As of v7.0, the “X avg days” label has been removed from Gate View cards (F-171). The value remains in the tooltip and is still used for sorting.
F-86 EXPLICIT Gate technicians display: If a Master Gates CSV is loaded, technician names are displayed in the gate view header after the order count, in parentheses, ordered by column priority (comma-separated). E.g. "10. PREP (3 orders) (John Smith, Jane Doe)". If the Master Gates file is loaded but a gate has no technician entries, the header shows "(No technicians assigned, check input files)". If no Master Gates file is loaded, no technician information appears.
F-87 EXPLICIT "Include Unmapped Operations" checkbox: A checkbox next to the Process button (checked by default) controls whether part numbers absent from the decoder ring are rendered in the Unmapped Operations section. When unchecked, unmapped data is not processed.
F-95 EXPLICIT Audit-CSV-driven placement: When a Mobile Audit CSV (Section 3.4) is loaded, Gate View placement is driven entirely by the audit data. The legacy algorithmic placement (F-60) is not used. Orders appear within each gate in the order they appear in the CSV file. When no audit CSV is loaded, the Gate View area displays a dashed-border placeholder message instead of gate cards.
F-96 EXPLICIT “Not found on a rack” section: When an audit CSV is loaded, any active work orders from the Daily Upload that are not present in the audit CSV are collected into a “Not found on a rack” section displayed after all numbered gates. This section uses an amber header and helps identify orders that may have been missed during the physical audit. The section is collapsible (F-120) and sorted (F-121).
F-97 EXPLICIT “Missing from daily upload”: If an order in the audit CSV is not found in the Daily Upload, its card displays an amber “Missing from daily upload” label instead of normal enrichment data. The card still appears on the gate it was audited on, using Part No. and Serial No. from the audit CSV if available.
F-98 EXPLICIT DR Operations indicator (Gate View): If an order has any active operations in the 7000–7999 range (DR Operations) that appear in the decoder ring, the Gate View card displays a red “DR Operations” label and a red 3px border to visually flag it for technicians.
F-99 EXPLICIT OPER STOP per-operation (Gate View): In Gate View card tooltips, individual operations whose order has “OPER STOP” or “OPER_HOLD” hold status show the hold text per operation line. On the card face itself, only order-level holds (“ORDER STOP” or “ORDER_HOLD”) are displayed (OPER STOP/OPER_HOLD is not shown on the card).
F-100 EXPLICIT Technician-bench display: If an audit CSV entry includes a Technician value, the Gate View card shows “with <tech name>” in italic muted text below the order details. Tech-bench orders are positioned first (far left) within their gate.
F-119 EXPLICIT QA Operations indicator (Gate View): If an order has any Active or In Queue operations where the Work Department ends in “_QA” (case-insensitive), the Gate View card receives a bright blue 3px border and a blue “QA Operations” label. Priority: The DR Operations red border and label (F-98) takes precedence — if an order has both DR Operations and QA operations, only the red border and “DR Operations” label is shown (no double borders, no double labels).
F-120 EXPLICIT “Not found on a rack” collapsible: The “Not found on a rack” section (F-96) is collapsible. Clicking the amber header toggles the card row open/closed. Default state: closed. A triangle icon in the header indicates the current state (▸ closed, ▾ open).
F-121 EXPLICIT “Not found on a rack” sort order: Cards in the “Not found on a rack” section are sorted by: (1) Program ascending, (2) orders with DR Operations first, (3) Part Number ascending, (4) Order Number ascending.
F-122 EXPLICIT Quick navigation links: After processing, two small text links appear near the top of the page: “Jump to Detail View” and “Jump to Unmapped Operations.” Each link opens its respective section (if not already open) and smooth-scrolls to it. Links are hidden in print/PDF output.
F-123 EXPLICIT Split Location Detection (Multi-Gate Orders): When processing the Audit CSV, the app detects work orders that appear in more than one gate. Both cards display an orange warning label showing which other gate(s) the order also appears on (e.g., “⚠ Also on Gate 5” or “⚠ Also on Hold Rack”). The tooltip also includes “SPLIT LOCATION: Also on [gate]” for additional visibility. This helps manufacturing engineers identify hardware that has been separated across racks and needs to be reunited.
F-124 EXPLICIT Mobile Audit — Already-Added Order Visibility: During mobile audit scanning, work orders that have already been added to a gate are no longer hidden from search results. Instead, they appear with an orange left border and a warning label showing which gate they are already on (e.g., “⚠ Already on Gate 5”). Users can still tap to add the order to the current gate, supporting legitimate multi-box scenarios where the same work order occupies multiple physical locations.
F-125 EXPLICIT Detail View — Audit Location Indicator: When an audit CSV is loaded, each work order in the Detail View displays its audit location below the S/N field. Three states: (1) Single gate — green text showing the gate label (e.g., “Gate 3” or “Hold Rack”); (2) Split location — orange warning badge with ⚠ icon listing all gates (e.g., “⚠ Gate 2, Hold Rack”), matching the Gate View multi-gate warning style; (3) Not found — amber badge reading “Not found at rack/tech bench” for orders present in the daily upload but absent from the audit CSV. When a technician is associated with the order in the audit CSV, the technician name is appended in parentheses (e.g., “Gate 3 (with John Smith)”). When no audit CSV is loaded, the indicator is hidden. Applies to both the main Detail View and the Unmapped Operations section.
F-126 EXPLICIT Detail View — order hold detection across all rows: The order hold flag in the Detail View must scan all operation rows for a given work order, not just the first row. If any row for the order has “ORDER STOP” or “ORDER_HOLD” in the Hold Status column, the red hold flag is displayed on the order card. This matches the Gate View behavior which already scanned all rows.
F-127 EXPLICIT Hold Rack collapsible: Each hold rack section in the Gate View is independently collapsible, using the same interaction pattern as “Not found on a rack” (F-120). Click the header to toggle open/closed. Default state: closed. Triangle icon indicates current state. Multiple hold racks (e.g., HOLD-1, HOLD-2) each get their own collapsible section with unique element IDs.
F-128 EXPLICIT Gate View — Show all known gates: The Gate View displays all gates from the Decoder Ring and the Master Gates file (if loaded), even if no work order cards are assigned to them. Empty gates show the gate header with “(0 orders)” and technician names (if loaded), followed by an empty card row. Gate names from the Master Gates file take priority; the Decoder Ring fills gaps for gates not in the Master Gates file. Hold-prefixed gates and decimal sub-gates are included. This provides a complete visual of the factory floor layout.
F-129 EXPLICIT Gate View — Legend: A legend is displayed above the Gate View gate sections (below the title). It shows: (1) a color swatch and label for each active program; (2) a yellow swatch labeled “Not found on daily upload”; (3) a blue-outlined swatch labeled “QA Operations”; (4) a red-outlined swatch labeled “DR Operations.” The legend is included in the saved snapshot.
F-130 EXPLICIT Snapshot — Navigation links and Unmapped Operations: The saved snapshot includes the “Jump to Detail View” and “Jump to Unmapped Operations” navigation links. It also includes the Unmapped Operations toggle button and view (if unmapped data exists), with full filter interactivity preserved.
F-131 EXPLICIT Master Gates file as gate location authority: When a Master Gates CSV is loaded, it becomes the authoritative source for which gates exist, appear in the Mobile Audit, and appear in the Gate View. All gates listed in the file appear as buttons in the Mobile Audit and as columns in the Gate View (even if empty). If no Master Gates file is loaded, the app falls back to deriving gates from the Decoder Ring (legacy behavior, no cross-validation). This separation allows gate locations to be defined independently of operation-to-gate mappings.
F-132 EXPLICIT Decimal sub-gates: Gate numbers may be decimal values (e.g., “7.1”) to represent sub-locations within a gate. Sub-gates sort numerically between their parent gates (7 → 7.1 → 8). Sub-gates appear as independent buttons in the Mobile Audit and independent columns in the Gate View. All sorting uses parseFloat rather than parseInt to preserve decimal ordering.
F-133 EXPLICIT Data-driven hold racks (HOLD-prefixed gates): Hold racks are no longer hardcoded. Any gate whose Gate Number starts with “HOLD” (case-insensitive) is treated as a hold rack — displayed with red styling, sorted before numbered gates, and independently collapsible in the Gate View (F-127). Multiple hold racks are supported (e.g., “HOLD-1”, “HOLD-2”), each with its own Gate Name. Hold racks are defined in the Master Gates CSV alongside regular gates. In the Mobile Audit, hold rack buttons show the Gate Name as their label (not the gate number).
F-134 EXPLICIT Gates without technicians: The Master Gates CSV may contain rows where all Technician columns are blank. These rows register the gate location (Gate Number + Gate Name) without assigning a technician. The gate appears in the Mobile Audit and Gate View but without technician buttons in the audit or technician labels in the Gate View header. This supports defining new gate locations before technician assignments are finalized.

8. Retrospective & Architecture Notes

This section reflects on the development process and identifies areas where, given the full picture from the start, the implementation could be more robust or efficient.

Observation 1: Monolithic process() function
The entire app logic—parsing both CSVs, matching, grouping, and rendering—lives in a single ~270-line process() function. If this were started fresh with all requirements known, it would benefit from separation into discrete steps: (1) parse & validate inputs, (2) build the data model, (3) render the view. This would make individual pieces testable and easier to extend (e.g., adding new filters or export features without touching rendering code). This separation would also directly support the eventual API migration by isolating the data-ingestion layer.
Observation 2: HTML string concatenation for rendering
The gate bar is built via html += '...' string concatenation. This works but is fragile—a missing quote or unescaped character can silently break the output. A lightweight template approach (even just a createElement-based builder, which is already used for outer containers) applied consistently would be safer and more maintainable.
Observation 3: Color logic could be data-driven
The chip color rules are hardcoded in an if/else chain. If the color scheme or the Work Dept values change, someone must edit JavaScript. An alternative: define a color-rule table at the top of the script (an array of {statusMatch, deptMatch, className} objects) and evaluate it in order. This would make the business rules visible and editable without understanding the surrounding code.
Observation 4: Decoder Ring column usage
(Partially resolved in v3.0) The Modifier column is now actively used—"ignore" causes an operation to be skipped during gate placement. The Decoder Ring's Work Dept column is still parsed but unused (the Daily Upload's Work Dept drives chip color logic). Consider either defining the Decoder Ring Work Dept's purpose or removing it from the expected format.
Observation 5: Incremental development vs. upfront design
Features were added one at a time (flat table → gate bar → compact cards → program grouping → color coding), which meant some work was done and then replaced. For example, the detail tables were built and then removed in favor of compact cards. Starting from a PRD like this one, with the visual layout and data model agreed upon first, would have avoided that rework. That said, the iterative process was valuable for discovering the actual requirements through hands-on experimentation, which is a legitimate approach—especially when the product owner is also the designer.
Observation 6: No automated testing
There are no unit tests for the CSV parser, the normOper() function, or the gate-grouping logic. Each of these has had bugs during development (leading zeros, gate merging, case sensitivity). Even a small inline test suite (runnable via a "Run Tests" button in the UI) would catch regressions when making changes.
Observation 7: Oper Status fallback behavior
(Now resolved in v2.0) The color logic previously defaulted unknown statuses to green, which could mislead users. Per product owner direction, any status other than "ACTIVE" or "IN QUEUE" now renders as grey. This is a safer default.
Observation 8: Snapshot uses Function.toString() for serialization
The Save Snapshot feature serializes interactive JavaScript functions using Function.prototype.toString() and re-attaches event listeners via delegation. This works reliably but means that any function referenced by the snapshot must be a named function declaration (not an arrow function assigned to a variable). If the codebase moves toward ES6+ patterns, the serialization approach may need updating.
Observation 9: Total Days column matching required multiple fallbacks
The "Total Days at Operation" column (column Z / 26th column) initially showed 0 for all cards. The fix required: (1) a fallback partial-match search if the exact column name doesn't match (handles embedded newlines from Excel), (2) a robust parseNum() helper that extracts numbers from strings even with unexpected formatting. This highlights the fragility of CSV column matching when files pass through Excel—a header validation step at load time could surface these issues earlier.
Observation 10: Audit-CSV architecture shift
The Gate View transitioned from an algorithmic placement model (walk the decoder ring, find the first active gate) to an audit-CSV-driven model (a physical rack audit determines placement). This is a fundamental architecture change that makes the Gate View a true reflection of physical reality rather than a computed approximation. The dailyLookup map pattern introduced for this—a pre-built map of orderNo → enrichment data (program, partNo, holdStatus, activeOps, maxDays, hasDrOps)—is cleaner than the previous inline computation approach and makes the rendering logic easier to follow.
Observation 11: Mobile Audit as a companion tool
The Mobile Audit generation (F-102–F-107) effectively makes the Gate Tracker a two-tool system: the desktop app processes and visualizes data, while the mobile audit tool captures physical-world state. The audit CSV closes the loop by feeding real observations back into the visualization. This pattern—generate a data-collection tool, then consume its output—could be extended to other use cases (e.g., quality inspections, material receiving).
Observation 12: Split-location detection driven by real-world bug
The split-location detection (F-123) and mobile audit already-added visibility (F-124) were driven by a real-world bug discovery: the mobile audit search was hiding already-scanned orders, causing auditors to manually re-enter work orders that occupied multiple physical boxes (e.g., the same W/O on Gate 2 and Gate 3). The root cause was the search filter that excluded already-added orders across all gates. Instead of simply removing the filter, the fix preserves visibility of already-added orders with a clear warning indicator, supporting the legitimate multi-box scenario while preventing accidental duplicates. This is a good example of how field usage reveals requirements that are invisible during desktop development.

10. Data & Security

EXPLICIT A security assessment was conducted (v4.0) covering the application file and PRD. The assessment combined findings from two independent AI-assisted reviews (ChatGPT and Claude Chat). This section documents the application's security posture, implemented controls, and remaining risks.

10.1 Threat Context

The Gate Tracker processes proprietary manufacturing data (work order numbers, part numbers, serial numbers, operational status, and program names). While the application runs entirely client-side with no network communication, the data it handles is sensitive and the resulting output files (snapshots, printed PDFs) may circulate within the organization.

10.2 Implemented Security Controls

IDSourceRequirement
S-01 EXPLICIT XSS prevention — HTML escaping: All user-supplied data rendered into HTML is passed through an esc() function that escapes &, <, >, ", and ' (ampersand, angle brackets, double quotes, single quotes). This prevents injection of executable HTML or JavaScript through CSV data.
S-02 EXPLICIT Safe DOM construction for errors: Error messages displayed when CSV columns are missing use document.createElement() + textContent instead of innerHTML. This eliminates any XSS vector in error paths.
S-03 EXPLICIT Content Security Policy (CSP): A <meta http-equiv="Content-Security-Policy" content="default-src 'self' 'unsafe-inline';"> tag is present in both the main application and snapshot output. This provides defense-in-depth by restricting resource loading to the same origin and inline scripts only — blocking external script injection, fetch to remote servers, and image/font loading from third-party domains.
S-04 EXPLICIT CSV field length limit: During CSV parsing, all field values are truncated to a maximum of 500 characters. This mitigates denial-of-service via excessively long fields (e.g., a maliciously crafted CSV with multi-megabyte cell values) and limits the blast radius of any data that does make it into the DOM.
S-05 EXPLICIT Proprietary data banner: Snapshot output files include a prominent "Proprietary — Internal Use Only" banner at the top of the page, visually alerting recipients that the file contains sensitive manufacturing data and should not be shared externally.
S-06 EXPLICIT Visible version number: The application displays a version number (v1.0) in the UI header. This provides traceability — when a snapshot or PDF is shared, the recipient can identify which version of the tool produced it, and security patches can be tracked.
S-07 ASSUMED No data transmission: The application makes zero network requests. No data is sent to any server, API, CDN, or analytics service. All processing occurs in the browser's JavaScript runtime. The CSP policy (S-03) enforces this at the browser level as an additional guarantee.
S-08 ASSUMED No persistent storage: The application does not use cookies, localStorage, sessionStorage, IndexedDB, or any other browser persistence mechanism. Closing the tab destroys all data. This eliminates data-at-rest risks in the browser.
S-09 ASSUMED Row count warning: A confirmation dialog warns the user if the daily upload exceeds 500 rows. While primarily a performance safeguard, this also serves as a data validation checkpoint — an unexpectedly large file may indicate the wrong data was loaded.
S-10 ASSUMED CSS selector injection prevention: All CSS selectors constructed with user-derived data (e.g., order numbers, program names) use CSS.escape() to sanitize values before interpolation into querySelector() calls. This prevents crafted CSV data from breaking out of attribute selectors.
S-11 ASSUMED Snapshot script tag sanitization: JSON data embedded in snapshot <script> blocks is sanitized by replacing < characters with \\u003c Unicode escapes. This prevents crafted CSV values containing </script> from prematurely terminating the script block and injecting HTML.

10.3 Remaining Risks & Unimplemented Recommendations

The following items were identified during the security assessment but have not yet been implemented. They are listed by priority and include a rationale for deferral.

IDPrioritySourceRisk & Recommendation
SR-01 Medium OPEN innerHTML still used for rendering: The gate view cards and detail view order cards are built via HTML string concatenation and injected with innerHTML. While all data values pass through esc(), this pattern is inherently riskier than DOM construction via createElement/textContent. A full refactor of the rendering pipeline to use safe DOM methods would eliminate this risk class entirely. Deferred because: the rendering logic builds complex nested HTML (gate bars, oper chips, tooltips) that would require a significant architectural rewrite. The esc() function provides adequate protection for the current threat model.
SR-02 Medium OPEN Snapshot data exposure: Snapshot HTML files contain the full dataset in cleartext. Anyone with access to the file can see all order numbers, part numbers, serial numbers, and program names. The "Proprietary" banner (S-05) is a visual deterrent but does not enforce access control. Recommendation: Organizational policy should govern snapshot file distribution (e.g., internal email only, no posting to public-facing systems). Technical encryption is impractical for a single-file HTML tool.
SR-03 Low OPEN No CSV schema validation beyond column headers: The application validates that required column names exist but does not validate data types, expected value ranges, or row structure. A malformed CSV could produce incorrect output without error. Recommendation: Add validation rules for critical fields (e.g., Order No. should be alphanumeric, Gate Number should be numeric, Oper Status should be from a known set).
SR-04 Low OPEN Regex DoS in parseNum(): The parseNum() fallback uses a simple regex (/[\d.]+/) to extract numbers from strings. For pathologically long input strings, this could theoretically be slow. Practical risk is very low because CSV fields are now capped at 500 characters (S-04).
SR-05 Low OPEN Function.toString() serialization: The snapshot feature serializes JavaScript functions using Function.prototype.toString(). This is non-standard and may break in future browser versions or non-mainstream JavaScript engines. No security risk, but a maintainability concern. Alternative: store function source as string literals.
SR-06 Low OPEN No audit logging: There is no record of who loaded which data, when processing occurred, or when snapshots were generated (beyond the snapshot's embedded timestamp). Acceptable for a single-user prototype. Would be required for a production deployment.
SR-07 Info ASSUMED No role-based access control: By design. The application is a local HTML file with no authentication or user management. Access control is handled at the file-system level (who has access to the HTML file and the CSV data). This is appropriate for a prototype; a production version would require integration with organizational identity management.
